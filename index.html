Watch Out

Common Pitfalls

🌫️

Being too vague in specs

"Build me an app" → garbage. Be specific about features, data, and user flows. The more context you give, the better the output.

💾

Not using version control

Without Git, one bad change can destroy hours of work. Commit early, commit often. It's your undo button.

🏔️

Trying to build too much at once

Start with the core feature. Get it working. Then add the next thing. Scope creep kills projects.

📱

Not testing on mobile

60%+ of your users are on phones. Test early. AI-generated layouts often need mobile refinement.

Connect Everything

🔌 MCP (Model Context Protocol)

Give Claude hands to reach into your world

What It Is

MCP is an open standard that lets Claude connect to external tools and data sources — databases, APIs, file systems, SaaS platforms. Instead of copy-pasting data into chat, Claude can go get it itself.

📦 Popular MCP Servers

  • • GitHub — read repos, create PRs, manage issues
  • • Postgres/Supabase — query your database directly
  • • Slack — read channels, send messages
  • • File System — read/write files on your machine
  • • Brave Search — web search from within Claude

⚠️ Context Cost

Each MCP server injects tool definitions into Claude's context window. 5 MCP servers can eat 10-15% of your context before you type a message.

Rule of thumb: Only enable the MCPs you need right now. Disable the rest.

Think of MCPs like browser extensions — powerful individually, but too many slows everything down.

Secret Weapon

💻 CLIs — Why LLMs Love the Command Line

LLMs Are Incredibly Good at CLI Tools

Command-line interfaces are text in, text out — exactly what LLMs are built for. Unlike GUIs that require clicking and navigating, CLIs let Claude read documentation, run commands, and parse output natively. It's the difference between describing a screenshot vs. reading a manual.

Why CLIs > GUIs for AI

  • • Deterministic — same command = same result
  • • Composable — pipe tools together for complex workflows
  • • Documentable — --help gives Claude everything it needs
  • • Scriptable — Claude can chain 10 commands in seconds
  • • No UI parsing — no screenshots, no pixel coordinates

Tools You Should Know About

  • • gh — GitHub from the terminal (PRs, issues, repos)
  • • vercel — deploy with one command
  • • supabase — manage your database locally
  • • npx/bunx — run any package without installing
  • • curl/jq — hit any API, parse any JSON

Pro tip: If a service has a CLI, tell Claude to use it. It'll be faster and more reliable than any API integration.

You're Always in Control

Claude Code asks before doing anything potentially destructive — deleting files, running shell commands, pushing to production. You approve or deny each action.

Permission Levels

  • 🟢 Auto-approve: Reading files, searching code — safe stuff
  • 🟡 Ask first: Writing files, running commands — needs your OK

Full Autopilot Mode

Run claude --dangerously-skip-permissions to let Claude run without asking for approval on every action. It'll read, write, execute, and deploy without stopping. Use with caution — great for experienced builders who trust the process, but start with permissions on until you're comfortable.

Software Engineering 101

🔐 Security Basics for Non-Engineers

You don't need a CS degree to ship secure software. But you do need to know a few rules. These apply whether you wrote the code or Claude did.

Environment Variables & Secrets

API keys, database passwords, and tokens go in a .env.local file — never in your code. This file is ignored by Git so it never gets uploaded to GitHub.

DATABASE_URL=postgres://...
STRIPE_SECRET_KEY=sk_live_...
NEXTAUTH_SECRET=random_string

If you accidentally push a key to GitHub, consider it compromised. Rotate it immediately. Also watch for TypeScript's process.env.KEY! — the ! suppresses the "might be undefined" warning but crashes at runtime if the var isn't set.

Row-Level Security (RLS)

If you use Supabase, RLS is your most important security layer. It ensures users can only read/write their own data — enforced at the database level, not just your UI.

  • • Always enable RLS on every table
  • • Write policies that check auth.uid()
  • • Test by trying to access another user's data
  • • Service role keys bypass RLS — keep them server-side only

⚠️ Real-world gotcha: Cron jobs & background tasks use the service role, where auth.uid() is NULL. Your RLS policies will silently block them unless you add a policy for auth.role() = 'service_role'.

HTTPS & Deployment

If you deploy with Vercel, Netlify, or Cloudflare Pages — HTTPS is automatic. Never deploy over plain HTTP. Set environment variables in your hosting dashboard, not in code.

The .gitignore File

This file tells Git what to never upload. Your .env.local, node_modules/, and any credentials should be listed here. Claude usually creates this for you — but always double-check.

Software Engineering 101

🔑 Authentication & Access Control

If your app has user accounts, you need auth. Don't build it from scratch — use a proven service. Here's the minimum you need to get right.

Use a Managed Auth Provider

Never roll your own login system. Use Supabase Auth, Clerk, or Auth0. They handle password hashing, session tokens, email verification, and OAuth — things that are easy to get wrong and dangerous when you do.

  • 🟢 Supabase Auth — Free tier, pairs with your database
  • 🟢 Clerk — Beautiful drop-in UI components
  • 🟢 Auth0 — Enterprise-grade, complex setups

Server vs. Client: Know the Difference

Your app has two sides. The client (browser) is visible to everyone — never trust it with secrets. The server is private — this is where sensitive operations happen.

  • • Anon key → safe for the browser (limited access)
  • • Service role key → server only (full access, bypasses RLS)
  • • API routes / server actions = your secure backend
  • • Middleware ≠ server — Supabase middleware clients use the anon key, not the service role. RLS still applies!

The Auth Checklist

  • ✅ Login + sign-up pages exist
  • ✅ Protected routes redirect unauthenticated users
  • ✅ RLS policies filter data by auth.uid()
  • ✅ Service role key is only used in server-side code
  • ✅ OAuth callback URLs configured in provider dashboard
  • ✅ Password reset flow works end-to-end

Common Auth Mistakes

These are real bugs found across multiple projects:

  • ❌ Hiding UI buttons instead of enforcing access server-side
  • ❌ Using the service role key in browser code
  • ❌ Querying "first user in database" instead of the actual authenticated user — a critical security bug that lets User A act as User B
  • ❌ Assuming middleware runs with service role permissions — it uses the anon key, so RLS still applies
  • ❌ No RLS policies — any user can read all data
  • ❌ Self-referencing RLS policies (e.g., checking is_admin on the same table being queried) — causes silent circular failures
  • ❌ Not testing what happens when a logged-out user hits a protected page
