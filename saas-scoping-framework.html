Phase 01

1.1 ‚Äî Core Features (MVP)

List only the absolute essential features needed to solve the core problem. For each one, define what it does, why it's essential, and how you'll know it's working.

1.2 ‚Äî User Journey Mapping

Phase 06

Risk Assessment & Mitigation Identify what could go wrong and plan for it

6.1 ‚Äî Technical Risks
  • Data loss ‚Üí Automated backups, point-in-time recovery
  • Security breach ‚Üí Auth best practices, input sanitization, rate limiting
  • Downtime ‚Üí Uptime monitoring, alerting, status page
  • Scaling bottlenecks ‚Üí DB indexing, caching strategy, horizontal scaling plan
  • Third-party failures ‚Üí Fallbacks for payment, email, storage providers
6.2 ‚Äî Business Risks
  • Low adoption ‚Üí Pre-launch waitlist, beta program, early feedback loops
  • High churn ‚Üí In-app feedback, exit surveys, engagement monitoring
  • Competition ‚Üí Clear differentiation, niche focus, switching costs
  • Regulatory / compliance ‚Üí GDPR, HIPAA, SOC 2 ‚Äî know what applies to you
  • Funding / runway ‚Üí Revenue targets before runway runs out

‚ñ∂ Best Practices

Working Effectively with Claude Code

üå±

01 ‚Äî Start Small

Begin with the simplest version of your core feature. Test and validate before adding complexity. Every feature you add is a feature to debug.

üéØ

02 ‚Äî Be Specific

Use concrete examples instead of abstract descriptions. Provide sample data. Include edge cases. "User-friendly" tells Claude nothing.

üë§

03 ‚Äî Think in User Stories

"As a [user type], I want to [action] so that [benefit]." This format gives Claude the context and purpose it needs.

‚ñ∂ Common Pitfalls

What to Avoid

Feature Creep

Trying to build everything at once. Every added feature multiplies complexity. Ship the core, validate, then expand.

Vague Requirements

"Make it user-friendly" without specifics. Claude will guess ‚Äî and guessing means revision cycles. Be precise.

Ignoring User Feedback

Building in isolation without showing real users. What you think they want and what they actually need often differ wildly.

Over-Engineering

Complex solutions for simple problems. Your MVP doesn't need microservices, real-time sync, or AI if users just need a form.

Under-Specifying

Not enough detail for Claude to execute. A thin spec leads to thin output. Garbage in, garbage out ‚Äî every time.

Skipping the Plan

Jumping straight to "build me an app" without this framework. 20 minutes of planning saves 20 hours of rework.
